🏠 Overview

This guide walks you through connecting a spoke device (MacBook, Raspberry Pi, Jetson, or any machine) to the SeaClip Hub. Once connected, the spoke can:

  • Receive coding tasks from the hub orchestrator
  • Run Agent Zero locally for autonomous coding
  • Use AI models hosted on the hub via Tailscale (Ollama: qwen3.5:35b, glm-4.7-flash)
  • Work on code in isolated git worktrees
  • Submit pull requests for hub review and merge
  • Report telemetry (CPU, RAM, disk) back to the hub dashboard
Architecture: The hub runs on Mac Studio M3 Ultra (96GB) with Ollama models, PGlite database, and the SeaClip dashboard. Spokes are thin clients that connect over Tailscale VPN and use the hub's API + models.

Step 1: Install & Connect Tailscale

Tailscale creates the encrypted mesh VPN between hub and spoke. If already installed, skip to verification.

macOS

# Install via Homebrew
brew install --cask tailscale

# Or download from: https://tailscale.com/download/mac

# Start and authenticate
open -a Tailscale
# Click "Log in" in the menu bar icon, use your rao.vinu@ account

Linux (Debian/Ubuntu)

curl -fsSL https://tailscale.com/install.sh | sh
sudo tailscale up

Verify Connection

# Check you can see the hub
tailscale status

# You should see:
# 100.114.140.33  whitenoises-mac-studio  rao.vinu@  macOS
# (your device)                           rao.vinu@  macOS

# Test connectivity to hub
ping -c 3 100.114.140.33

# Test API access
curl -s http://100.114.140.33:3001/health
# Should return: {"status":"ok","version":"0.1.0",...}

# Test Ollama access
curl -s http://100.114.140.33:11434/api/tags | python3 -m json.tool
# Should show qwen3.5:35b and glm-4.7-flash
Important: If ping 100.114.140.33 fails, check that both devices are logged into the same Tailscale account (rao.vinu@) and that the hub's firewall allows Tailscale traffic.

Registering a NEW spoke device

# Set your hub address
HUB="http://100.114.140.33:3001"
COMPANY_ID="4ba03a16-90db-4acc-86e4-943f943db118"

# Get your Tailscale IP
MY_IP=$(tailscale ip -4)
MY_HOSTNAME=$(hostname)

# Register with the hub
curl -X POST "$HUB/api/companies/$COMPANY_ID/edge-devices" \
  -H "Content-Type: application/json" \
  -d "{
    \"name\": \"$MY_HOSTNAME (Spoke)\",
    \"deviceType\": \"mac\",
    \"ipAddress\": \"$MY_IP\",
    \"capabilities\": [\"agent-zero\", \"coding\"],
    \"metadata\": {
      \"hostname\": \"$MY_HOSTNAME\",
      \"tailscaleIp\": \"$MY_IP\",
      \"role\": \"spoke\"
    }
  }"

# Save the returned device ID — you'll need it for all subsequent commands
# Example response: {"id":"abc12345-...","name":"my-host (Spoke)",...}

Save your device config

Step 5: Start Telemetry Reporting

The telemetry script sends CPU/RAM/disk usage to the hub every 30 seconds, keeping your device status green on the dashboard.

# Create the telemetry script
mkdir -p ~/.seaclip
cat > ~/.seaclip/telemetry.sh << 'SCRIPT'
#!/bin/bash
# SeaClip Spoke Telemetry Reporter
# Sends system metrics to the hub every 30 seconds

source ~/.seaclip/spoke.env

while true; do
  # Gather metrics (macOS)
  CPU=$(ps -A -o %cpu | awk '{s+=$1} END {printf "%.1f", s/4}')
  MEM=$(memory_pressure 2>/dev/null | grep "System-wide memory free percentage" | awk '{print 100 - $NF}' || echo "50")
  DISK=$(df -h / | tail -1 | awk '{print $5}' | tr -d '%')
  # Temperature (macOS, requires osx-cpu-temp or similar)
  TEMP=$(which osx-cpu-temp >/dev/null 2>&1 && osx-cpu-temp 2>/dev/null | grep -o '[0-9.]*' | head -1 || echo "0")

  # Send to hub
  curl -s -X POST \
    "$SEACLIP_HUB_URL/api/companies/$SEACLIP_COMPANY_ID/edge-devices/$SEACLIP_DEVICE_ID/telemetry" \
    -H "Content-Type: application/json" \
    -d "{
      \"cpuPercent\": $CPU,
      \"memoryPercent\": ${MEM:-50},
      \"diskPercent\": $DISK,
      \"temperatureCelsius\": ${TEMP:-0},
      \"uptimeSeconds\": $(( $(date +%s) - $(sysctl -n kern.boottime | awk '{print $4}' | tr -d ',') )),
      \"timestamp\": \"$(date -u +%Y-%m-%dT%H:%M:%SZ)\"
    }" > /dev/null 2>&1

  echo "[$(date '+%H:%M:%S')] Telemetry sent: CPU=${CPU}% MEM=${MEM}% DISK=${DISK}%"
  sleep 30
done
SCRIPT
chmod +x ~/.seaclip/telemetry.sh

Step 6: Pull Tasks from Hub

The hub assigns tasks to your device. You can poll for new tasks.

# Check assigned tasks
source ~/.seaclip/spoke.env
curl -s "$SEACLIP_HUB_URL/api/spoke/$SEACLIP_DEVICE_ID/tasks" | python3 -m json.tool

# Check spoke-specific tasks (git workflow tasks)
curl -s "$SEACLIP_HUB_URL/api/spoke/$SEACLIP_DEVICE_ID/spoke-tasks" | python3 -m json.tool

Auto-poll script

# Create task poller
cat > ~/.seaclip/poll-tasks.sh << 'SCRIPT'
#!/bin/bash
source ~/.seaclip/spoke.env

echo "Polling for tasks from hub..."
while true; do
  TASKS=$(curl -s "$SEACLIP_HUB_URL/api/spoke/$SEACLIP_DEVICE_ID/spoke-tasks" 2>/dev/null)
  COUNT=$(echo "$TASKS" | python3 -c "import json,sys; d=json.load(sys.stdin); print(d.get('count',0))" 2>/dev/null)

  if [ "$COUNT" != "0" ] && [ -n "$COUNT" ]; then
    echo ""
    echo "========================================="
    echo "  $COUNT task(s) assigned!"
    echo "========================================="
    echo "$TASKS" | python3 -c "
import json, sys
data = json.load(sys.stdin).get('data', [])
for t in data:
    print(f\"  [{t.get('status','?')}] {t.get('title','?')}\")
    if t.get('branch'):
        print(f\"         branch: {t['branch']}\")
"
  fi

  sleep 60
done
SCRIPT
chmod +x ~/.seaclip/poll-tasks.sh

Step 7: Git Worktree Workflow

Each task gets its own git worktree — an isolated copy of the repo so you can work on multiple tasks without branch conflicts.

Initial repo setup (once per spoke)

# Clone the main repo
cd ~
git clone git@github.com:t4tarzan/seaclip-v1.git seaclip-v1
cd seaclip-v1

# Create worktrees directory
mkdir -p ~/.seaclip/worktrees

Start working on a task

# When you receive a task, create a worktree for it
TASK_BRANCH="spoke/task-123-fix-dashboard"

# Create worktree
git worktree add ~/.seaclip/worktrees/$TASK_BRANCH -b $TASK_BRANCH origin/main

# Work in the worktree
cd ~/.seaclip/worktrees/$TASK_BRANCH

# Install dependencies
pnpm install

# Tell the hub you started
source ~/.seaclip/spoke.env
curl -s -X PATCH \
  "$SEACLIP_HUB_URL/api/companies/$SEACLIP_COMPANY_ID/spoke-tasks/TASK_ID/status" \
  -H "Content-Type: application/json" \
  -d '{"status": "in_progress"}'

Use Agent Zero to work on the task

# From the worktree directory
cd ~/.seaclip/worktrees/$TASK_BRANCH

# Start Agent Zero pointed at this directory
cd ~/agent-zero
source venv/bin/activate
python main.py

# In Agent Zero, give it the task:
# > Working directory is ~/.seaclip/worktrees/spoke/task-123-fix-dashboard
# > Task: Fix the dashboard metric cards to show correct values
# > The code is a React + TypeScript project using TailwindCSS

Commit your work

cd ~/.seaclip/worktrees/$TASK_BRANCH
git add -A
git commit -m "fix: dashboard metric cards display correct values

Spoke: whites-macbook-air
Task: SC-123"
git push origin $TASK_BRANCH

Step 8: Submit Pull Request

Once your work is done, submit a PR for the hub to review.

# Push branch and create PR via GitHub CLI
cd ~/.seaclip/worktrees/$TASK_BRANCH
git push -u origin $TASK_BRANCH

gh pr create \
  --title "fix: dashboard metric cards" \
  --body "## Task SC-123
- Fixed metric card value mapping
- Spoke: whites-macbook-air

## Test
- Dashboard loads without errors
- All 4 metric cards show correct data"

# Notify the hub
source ~/.seaclip/spoke.env
TASK_ID="your-task-id-here"

curl -s -X POST \
  "$SEACLIP_HUB_URL/api/companies/$SEACLIP_COMPANY_ID/pull-requests" \
  -H "Content-Type: application/json" \
  -d "{
    \"spokeTaskId\": \"$TASK_ID\",
    \"deviceId\": \"$SEACLIP_DEVICE_ID\",
    \"title\": \"fix: dashboard metric cards\",
    \"description\": \"Fixed metric card value mapping\",
    \"sourceBranch\": \"$TASK_BRANCH\",
    \"targetBranch\": \"main\"
  }"

# Update task status
curl -s -X PATCH \
  "$SEACLIP_HUB_URL/api/companies/$SEACLIP_COMPANY_ID/spoke-tasks/$TASK_ID/status" \
  -H "Content-Type: application/json" \
  -d '{"status": "pr_raised"}'

Cleanup after merge

# After the hub merges your PR:
cd ~/seaclip-v1
git worktree remove ~/.seaclip/worktrees/$TASK_BRANCH
git branch -d $TASK_BRANCH

🔧 Troubleshooting

Can't reach hub (100.114.140.33)

  1. Check Tailscale is running: tailscale status
  2. Check you're on the same Tailnet: look for rao.vinu@ account
  3. Try tailscale ping 100.114.140.33
  4. Check hub firewall: sudo pfctl -s rules (macOS) — Tailscale traffic should be allowed
  5. Restart Tailscale: sudo tailscale down && sudo tailscale up

Ollama model not responding

# Check if Ollama is running on hub
curl -s http://100.114.140.33:11434/api/tags

# If connection refused, SSH to hub and restart:
# ssh whitenoise-oc@100.114.140.33
# ollama serve &

# Check model is loaded
curl -s http://100.114.140.33:11434/api/tags | grep qwen

Telemetry not showing on dashboard

# Test telemetry endpoint manually
source ~/.seaclip/spoke.env
curl -v -X POST \
  "$SEACLIP_HUB_URL/api/companies/$SEACLIP_COMPANY_ID/edge-devices/$SEACLIP_DEVICE_ID/telemetry" \
  -H "Content-Type: application/json" \
  -d '{"cpuPercent": 25, "memoryPercent": 60, "diskPercent": 45}'

# Should return 202 with {"accepted": true, ...}
# If 404: your device ID is wrong, re-check spoke.env

❓ Frequently Asked Questions

Do I need Ollama installed on the spoke?

No. The spoke uses the hub's Ollama over Tailscale. This means all AI inference happens on the Mac Studio's M3 Ultra GPU. The spoke just sends HTTP requests to 100.114.140.33:11434.

Can I use Claude Code on the spoke?

Claude Code requires its own API key and runs locally. It does NOT proxy through the hub. If you have a Claude API key, install it normally on the spoke. Otherwise, use Agent Zero which routes through the hub's Ollama.

How do I add a new device (Raspberry Pi, Jetson, etc.)?

Same process: (1) install Tailscale, (2) register via the API with deviceType set to raspberry_pi or jetson, (3) start telemetry. For ARM devices, Agent Zero might need different setup but the SeaClaw C binary is designed exactly for these thin-client edge devices.

What happens if the hub goes offline?

The spoke loses access to AI models and can't report telemetry. However, any local work (git worktrees, code changes) is safe. When the hub comes back, telemetry resumes automatically and you can push your branches.

Can multiple spokes work on the same repo?

Yes! Each spoke works in its own git worktree on a separate branch. There are no conflicts. The hub merges PRs one at a time through the dashboard.

How fast is Ollama over Tailscale?

On the same LAN: ~50-100ms additional latency vs local. Over WAN (different networks): 200-500ms. The qwen3.5:35b model generates ~20-30 tokens/sec on the M3 Ultra, so the network overhead is minimal compared to generation time.

How do I see my device on the dashboard?

Visit http://100.114.140.33:5173/edge-mesh to see all devices. Your device-specific view is at http://100.114.140.33:5173/spoke/YOUR_DEVICE_ID.

What's the difference between "tasks" and "spoke tasks"?

Tasks (/api/spoke/:id/tasks) are regular issues assigned to agents on your device. Spoke tasks (/api/companies/:id/spoke-tasks) are the git workflow tasks specifically designed for the spoke-based PR process — they track branch, worktree, and PR status.

How do I update my device's capabilities? 
curl -X PATCH \
  "$SEACLIP_HUB_URL/api/companies/$SEACLIP_COMPANY_ID/edge-devices/$SEACLIP_DEVICE_ID" \
  -H "Content-Type: application/json" \
  -d '{"capabilities": ["agent-zero", "coding", "testing"]}'
Can I use a different AI framework instead of Agent Zero?

Absolutely. Any tool that can call an OpenAI-compatible API works. Point it at http://100.114.140.33:11434/v1 (Ollama's OpenAI-compatible endpoint). Examples: LangChain, CrewAI, AutoGen, or even raw curl/Python scripts.
