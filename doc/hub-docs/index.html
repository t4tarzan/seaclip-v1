🏠 Welcome to Hub Command Center

Imagine an airport. Your Mac Studio is the main terminal building. Every device you own — phones, Raspberry Pis, Jetson Nanos, cameras — is a gate at that airport. Tailscale is the network of runways connecting everything. And tmux is the flight control tower, letting you see and manage every flight at once.

This documentation covers how to run the whole airport.

What is this?

Hub Command Center turns your Mac Studio M3 Ultra (96GB RAM) into a central hub for:

  • AI development — Run local models with Ollama, orchestrate Claude Code agents
  • Edge device management — Connect and monitor phones, Pis, Nanos, cameras
  • Team collaboration — Share tmux sessions, dashboards, and services over Tailscale
  • Everything-in-one-place — One terminal, seven windows, zero tab chaos

⚡ Quick Start

You just want this working. No reading, no philosophy, just go. Here's the fastest path from zero to hub.

5-minute path
If you already have Homebrew installed, you can be up and running in about 5 minutes. Just follow the checkboxes below in order.
  • 1. Install Homebrew (if missing)
    The package manager for macOS. Everything else installs through it.
  • 2. Run the hub installer
    This installs tmux, Docker, Ollama, Tailscale, Claude Code, and more.
  • 3. Run the preflight check
    Makes sure everything installed correctly and services are reachable.
  • 4. Start Docker and Tailscale
    These are GUI apps. Open them once so they run in the background.
  • 5. Start Ollama
    The local AI model server. It runs quietly in the background.
  • 6. Launch the hub
    This creates the full 7-window tmux session. You're in the airport now.
  • 7. Open the web dashboard (optional)
    A web UI you can view from any device on your network.

Commands for each step

Step 1: Homebrew

bash
/bin/bash -c "$(curl -fsSL https://raw.githubusercontent.com/Homebrew/install/HEAD/install.sh)"

Step 2: Hub installer

bash
~/hub/bin/hub-install.sh

This runs brew install for: tmux, gitmux, git, gh, jq, node, watch, ollama, Docker, Tailscale, and Claude Code.

Step 3: Preflight check

bash
hub-check

This checks every tool, service, auth token, and config file. Green checkmarks mean you're good. Red X's mean you have work to do.

Step 4: Start Docker and Tailscale

bash
open -a Docker
open -a Tailscale

Step 5: Start Ollama

bash
ollama serve

Step 6: Launch the hub

bash
hub

That's it. You're in. You should see the dashboard in window 1 with a big status board. Press Ctrl+a then 2 to jump to the Claude Code window.

Step 7: Web dashboard (optional)

bash
hub-dash

Open http://localhost:9090 in your browser. Or use your Mac's IP to access it from other devices.

⌨ Hub Commands

These are all the commands you get out of the box. They're shell aliases (shortcuts) defined in your ~/.zshrc file. Think of them as speed dial buttons for the hub.

Hub Commands

CommandWhat it doesDetails

hub-check Run the preflight check Validates all tools, services, and configs. Run this when something feels off.

▣ tmux Layout Guide

When you type hub, it creates a tmux session with 7 windows. Think of each window like a different desk in a control room. Each desk has a specific job.

The 7 Windows

┌───────────────────────────────────────────────────────┐ │ 1:dashboard 2:claude 3:git 4:services 5:network 6:logs 7:scratch │ └───────────────────────────────────────────────────────┘ Window 1: DASHBOARD Window 4: SERVICES (split into 2 panes) ┌─────────────────────┐ ┌─────────────────────┐ │ hub-status.sh │ │ Docker / services │ │ (auto-refreshes) │ │ monitor │ │ every 10 seconds │ ├─────────────────────┤ │ │ │ Ollama / local │ │ │ │ models │ └─────────────────────┘ └─────────────────────┘
WindowNameWhat it's for
1dashboardLive status board. Shows system resources, services, network, git repos. Auto-refreshes every 10 seconds.
2claudeYour main Claude Code agent. This is where you do the actual AI-assisted work.
3gitGit operations. Check status, stage, commit, push, review PRs. Keep your git work separated from Claude.
4servicesSplit into two panes. Top: Docker and service management. Bottom: Ollama and local model management.
5networkTailscale status, device monitoring, network diagnostics.
6logsOpens in ~/hub/logs. Tail log files, debug issues, watch output.
7scratchYour "junk drawer" terminal. Quick commands, experiments, one-off scripts. No judgment here.

Commands for Each Window

Here's what you can do in each window. Switch windows with Ctrl+a, release, then press the number.

Important: Two-step keys
tmux keybindings are a two-step sequence. Press Ctrl+a together, let go of both keys, then press the next key. It's NOT all at once.

Window 1: Dashboard — Ctrl+a then 1

Auto-refreshing status board. Just watch it. If you need to stop it: Ctrl+c, then restart with:

bash
watch -n 10 -c ~/hub/bin/hub-status.sh

Window 2: Claude Code — Ctrl+a then 2

Your AI agent. Claude Code starts automatically. Common things to ask it:

text
# Inside Claude Code, just type naturally:
"Show me all open PRs"
"Fix the bug in auth.ts and commit"
"Create a new branch, implement JWT auth, open a PR"
"Run the tests and fix any failures"
"Explain what this codebase does"

Window 3: Git & PRs — Ctrl+a then 3

For manual git work. First cd into your project, then use these:

bash
# First, go to your project
cd ~/your-project

# Basic git
git status                        # what's changed?
git log --oneline -10             # recent commits
git diff                          # see changes
git add . && git commit -m "msg"  # stage and commit
git push                          # push to remote

# GitHub PRs (via gh CLI)
gh pr list                        # list open PRs
gh pr view 42                     # view PR #42
gh pr create                      # create a new PR
gh pr checkout 42                 # checkout a PR locally
gh pr merge 42                    # merge a PR
gh pr review 42 --approve         # approve a PR

# Branches
git branch -a                     # list all branches
git checkout -b new-feature       # create new branch
git switch main                   # switch to main

Window 4: Services — Ctrl+a then 4

Split into two panes (top = Docker, bottom = Ollama). Switch between panes: Ctrl+a then j (down) or k (up).

bash
# ── Top pane: Docker ──
docker ps                                    # running containers
docker ps -a                                 # all containers (including stopped)
docker compose up -d                         # start services from docker-compose.yml
docker compose down                          # stop services
docker logs -f container_name                # follow logs
docker stats                                 # live resource usage
docker system prune                          # clean up unused stuff

# ── Bottom pane: Ollama ──
ollama list                                  # installed models
ollama ps                                    # currently loaded models
ollama run qwen3:30b-a3b                     # chat with a model
ollama pull llama3.2                         # download a model
ollama rm model_name                         # delete a model
curl http://localhost:11434/api/tags | jq    # API: list models
curl http://localhost:11434/api/generate \
  -d '{"model":"qwen3:30b-a3b","prompt":"hello"}'  # API: generate

Window 5: Network — Ctrl+a then 5

bash
# Local network

# Tailscale

# See who's hitting your services

lsof -i :9090                        # who's using dashboard port

Window 6: Logs — Ctrl+a then 6

bash

How to Navigate

The fastest way: Ctrl+a (release) then the window number. For example, Ctrl+a (release) 2 takes you to Claude.

To go next/previous: Ctrl+a Ctrl+l (next) or Ctrl+a Ctrl+h (previous).

Splitting Panes

Any window can be split into multiple panes. Think of it like splitting a desk in half.

  • Ctrl+a | — Split side by side (vertical divider)
  • Ctrl+a - — Split top and bottom (horizontal divider)
  • Navigate between panes: Ctrl+a then h/j/k/l (vim-style)
  • Resize panes: Ctrl+a then H/J/K/L (capital = resize)

iTerm2 Users: Control Mode

iTerm2 tip
If you use iTerm2, the hub automatically detects it and launches tmux in control mode (-CC). This turns every tmux window into a native iTerm2 tab, and panes become native splits. It looks and feels like a normal iTerm2 setup but with all the power of tmux underneath.

To use control mode manually:

bash
tmux -CC attach -t hub

Dev Session (Lighter Layout)

Don't need the full 7-window hub? The start-dev.sh script creates a lighter 4-window layout for project work:

start-dev.sh [project-name] [project-path] Window 1: claude ← Claude Code agent, focused on your project Window 2: git ← Git status and operations Window 3: server ← Run your dev server, watch output Window 4: scratch ← Quick one-offs

⚙ Services

The hub relies on four main services. Think of them like the utilities in a building — electricity (Docker), the kitchen (Ollama), the phone system (Tailscale), and the smartest employee (Claude Code).

🐳 Docker

Docker runs apps in containers (little isolated boxes). You use it for databases, web servers, dev tools — anything that needs its own environment without messing up your Mac.

Start / Stop

bash
# Start Docker (it's a GUI app on Mac)
open -a Docker

# Check if it's running
docker info

# See running containers
docker ps

# Stop all containers
docker stop $(docker ps -q)

# Nuclear option: stop Docker entirely
osascript -e 'quit app "Docker"'
Good to know
Docker Desktop on Mac uses a lightweight Linux VM behind the scenes. It takes about 2GB of RAM when idle. If you're tight on memory, you can quit it when you don't need containers.

🤖 Ollama

Ollama runs AI models locally on your Mac. No internet needed, no API costs, your data stays on your machine. With 96GB of RAM and an M3 Ultra, you can run some seriously capable models.

Start / Stop

bash
# Start the Ollama server
ollama serve

# In another terminal, pull a model
ollama pull llama3.1
ollama pull codellama
ollama pull mistral

# List your downloaded models
ollama list

# Chat with a model
ollama run llama3.1

# See what's loaded (API call)
curl http://localhost:11434/api/tags | python3 -m json.tool

Ollama runs on port 11434 by default. The hub status board checks this port to know if Ollama is alive.

🌐 Tailscale

Tailscale creates a private network (called a mesh VPN) between all your devices. Imagine a secret tunnel system connecting your Mac, phone, Raspberry Pi, and your teammate's laptop — all encrypted, all peer-to-peer.

Start / Stop

bash
# Start Tailscale (GUI app)
open -a Tailscale

# Check status
tailscale status

# Get your Tailscale IP
tailscale ip -4

# See connected devices
tailscale status --json | python3 -m json.tool

# Share a port with your team (e.g., the dashboard)
tailscale serve 9090
Why Tailscale?
Normal networks are annoying. NAT, firewalls, dynamic IPs, port forwarding — it's all a headache. Tailscale makes every device directly reachable with a stable IP. Your phone can SSH into your Mac from a coffee shop. Your teammate can reach your dashboard from their house. It just works.

🧠 Claude Code

Claude Code is an AI coding agent that runs in your terminal. It can read files, write code, run commands, create git branches, and even spawn other agents. It's like having a very capable intern who never sleeps and never complains about code reviews.

Basic Usage

bash
# Start Claude Code
claude

# Start in a specific directory
cd ~/myproject && claude

# Check version
claude --version

Model Strategy

Claude comes in three sizes, like T-shirts. Use the right one for the job:

ModelRoleBest forCost
Opus The Architect Complex design decisions, tricky debugging, system architecture $$$
Sonnet The Builder Writing code, implementing features, refactoring, most daily work $$
Haiku The Worker Quick edits, formatting, simple questions, bulk operations $

Think of it like a restaurant kitchen. Opus is the head chef who designs the menu. Sonnet is the line cook who makes the dishes. Haiku is the prep cook who chops the onions.

◉ Web Dashboard

The web dashboard is a live status page you can open in any browser. It's the "flight departures board" for your hub — shows every service, device, and repo at a glance.

How to Start

bash
# Start the dashboard
hub-dash

# Or directly:
python3 ~/hub/dashboard/server.py

It starts a Python web server on port 9090.

How to Access

FromURL
Same Machttp://localhost:9090
Same WiFihttp://<mac-local-ip>:9090
Tailscalehttp://<mac-tailscale-ip>:9090

Not sure what your IP is? Run ipconfig getifaddr en0 for local, or tailscale ip -4 for Tailscale.

What it Shows

The dashboard auto-refreshes every 10 seconds and displays:

  • System stats — CPU usage, memory (out of 96GB), disk space, uptime
  • Services — Docker (how many containers), Ollama (how many models), Tailscale (connected?), Claude Code (running?)
  • Network — Local IP, Tailscale IP, all connected devices with online status
  • Git repos — Name, current branch, number of uncommitted changes
  • tmux sessions — All active sessions
  • Quick reference — Common commands and keybindings

API Endpoint

The dashboard also exposes a JSON API. You can hit it from scripts, other tools, or monitoring systems:

bash
# Get the full status as JSON
curl http://localhost:9090/api/status | python3 -m json.tool

This returns everything: system stats, service statuses, device list, repo info, and tmux sessions. Handy for building your own integrations.

📡 Edge Devices

Edge devices are any computer or gadget that lives "out there" — not on your desk, but still part of your network. Phones, Raspberry Pis, Jetson Nanos, security cameras, sensors. Your Mac Studio is the brain; these are the hands and eyes.

Connecting via Tailscale

Every device connects to your hub through Tailscale. Once connected, they can talk to each other as if they're on the same local network — even if one is at home and the other is at a coffee shop.

Raspberry Pi

bash — on the Pi
# Install Tailscale
curl -fsSL https://tailscale.com/install.sh | sh

# Connect to your network
sudo tailscale up

# Verify connection
tailscale status

# Now you can SSH from your Mac:
# ssh pi@<pi-tailscale-ip>

iPhone / Android

  1. Install the Tailscale app from the App Store / Play Store
  2. Sign in with the same account you use on your Mac
  3. Toggle the VPN on
  4. Now you can access http://<mac-tailscale-ip>:9090 from your phone's browser to view the dashboard

Jetson Nano

bash — on the Jetson
# Install Tailscale (same as any Linux)
curl -fsSL https://tailscale.com/install.sh | sh
sudo tailscale up

# Run inference on the edge, report back to hub
# The Nano handles the ML, the Mac stores the results

IP Cameras

Many IP cameras can't run Tailscale directly. Instead, put a Raspberry Pi on the same network as the camera, install Tailscale on the Pi, and use the Pi as a bridge. The Pi grabs the camera's RTSP stream and forwards it to your hub.

Use Cases

📷 Monitoring

Camera feeds piped through Pis. View from your Mac or phone via Tailscale. Store recordings centrally.

🤖 Edge Inference

Run lightweight models on Jetson Nanos for real-time detection. Send results to your Mac for aggregation.

📈 Data Collection

Sensors on Pis send data to your hub. Your Mac processes, stores, and visualizes it. No cloud needed.

📱 Mobile Access

Check your hub dashboard from your phone anywhere in the world. Tailscale makes it feel like you're on the same WiFi.

👥 Team Workflow

Your hub isn't just for you. Team members can connect to it, share sessions, and collaborate in real time. Think of it like a shared control room — everyone can see the same screens.

How Team Members Connect

  1. They install Tailscale and join your network (you invite them from the Tailscale admin console)
  2. They SSH into your Mac Studio using the Tailscale IP
  3. They attach to a shared tmux session
bash — teammate's machine
# SSH into the hub
ssh youruser@100.x.x.x

# Attach to the same tmux session (shared view!)
tmux attach -t hub

# Or view the dashboard from their browser
open http://100.x.x.x:9090

Shared tmux Sessions

When two people attach to the same tmux session, they see the exact same thing. It's like pair programming but with zero screen sharing lag. You can also create separate sessions for parallel work:

bash
# Create a session for a specific task
tn feature-auth    # teammate works on auth
tn feature-ui      # you work on UI

# List all active sessions
tl

Git Workflow

Claude Code can manage your git workflow end to end. Here's the pattern that works well with teams:

  1. Create a branch for the feature/fix
  2. Let Claude Code do the work in the claude window
  3. Review changes in the git window
  4. Claude Code can create the PR directly using the GitHub CLI (gh)
  5. Team reviews the PR in GitHub
bash
# In the Claude window, you might say:
# "Create a new branch called fix-login-bug, fix the login
#  validation issue, and create a PR with a good description."
#
# Claude Code will:
# 1. git checkout -b fix-login-bug
# 2. Edit the files
# 3. git add, git commit
# 4. gh pr create --title "Fix login validation" --body "..."

Communication Patterns

With the hub setup, teams typically work like this:

  • Synchronous — Both in the same tmux session, pair programming. One types, both see it.
  • Asynchronous — Leave Claude Code running a task. Teammate checks dashboard later to see status. Review PRs when ready.
  • Hybrid — Quick Slack/Discord message: "check window 3, I left you something." Teammate SSHs in and sees it.

🐝 Agent Swarm Guide

An agent swarm is when you ask Claude Code to split work across multiple agents. Instead of one agent doing everything sequentially, you get a team of specialists working in parallel. It's like having an entire dev team instead of one developer.

What is a Swarm?

Normally, Claude Code is one agent in one terminal. A swarm means multiple Claude Code instances running at the same time, each in its own tmux pane or window, each working on a different piece of the puzzle.

How it works
You tell the "lead" agent what you want. It plans the work, splits it into subtasks, and spawns helper agents for each one. Each helper works independently, then the lead combines the results. Like an orchestra conductor directing different sections.

Starting a Swarm

bash
# Quick start with the swarm alias
swarm

1. Backend API routes (auth, tokens, middleware)
2. Database schema and migrations
3. Frontend login/signup components
4. Tests for all of the above

prompt

- Agent 2: Update all imports to use the new path aliases

- Agent 4: Write missing tests

Bug investigation

prompt
Users are reporting slow page loads. Investigate:
- Agent 1: Profile the API endpoints, find slow queries
- Agent 2: Check the frontend bundle size and loading waterfall
- Agent 3: Review recent commits for performance regressions

Report findings and suggest fixes.

Best Practices

  • Be specific about boundaries. Tell each agent exactly which files or directories to work in. "Agent 1 works in src/api/, Agent 2 works in src/ui/." This prevents merge conflicts.
  • Use Sonnet for most work. Only escalate to Opus when you need architectural decisions or complex debugging. Use Haiku for formatting, linting, and simple edits.
  • Check in after each phase. Don't let agents run for 30 minutes without reviewing. Check their output, correct course if needed.
  • Keep a scratch window open. Window 7 is your friend. Use it to verify what agents are doing — check git status, run tests, look at files.

🔧 Troubleshooting

Something broke? Don't panic. The airport has emergency procedures. Here are the most common issues and how to fix them.

Step Zero: Run the Preflight Check

Before debugging anything specific, run this. It checks everything at once:

bash
hub-check

Green checkmarks = good. Red X = problem. Yellow warning = works but not ideal.

"Service Won't Start" Checklist

Docker won't start?
  1. Is the Docker app actually open? open -a Docker
  2. Wait 30 seconds. Docker is slow to start on Mac.
  3. Check: docker info. If it says "Cannot connect," Docker Desktop isn't fully started yet.
  4. Still stuck? Quit Docker, reopen it. Sometimes it just needs a restart.
Ollama won't start?
  1. Is something else using port 11434? lsof -i :11434
  2. Try: ollama serve in a terminal and read the error message.
  3. Kill and restart: pkill ollama && ollama serve
Tailscale won't connect?
  1. Is the Tailscale app open? open -a Tailscale
  2. Check: tailscale status. Look for errors.
  3. Log in again: tailscale login
  4. Check your network. Tailscale needs internet to coordinate (but traffic flows peer-to-peer after that).

"Can't Connect From Device" Checklist

  1. Is the device on Tailscale? Check tailscale status on both machines.
  2. Can you ping the hub? ping <hub-tailscale-ip>
  3. Is the service running on the hub? Check the dashboard or run hub-check.
  4. Is the port accessible? The dashboard runs on 9090. curl http://<hub-ip>:9090
  5. Firewall blocking? macOS sometimes blocks incoming connections. Check System Settings > Network > Firewall.

How to Reset Everything

The nuclear option. When nothing else works, tear it all down and rebuild:

bash
# Kill all tmux sessions
tmux kill-server

# Stop all Docker containers
docker stop $(docker ps -q) 2>/dev/null

# Kill Ollama
pkill ollama

# Now start fresh
hub
